Binance 1m OHLCV 데이터 컬럼 정리
=================================

대상 데이터:
- binance_1m_ohlcv/BTCUSDT_1m_ohlcv.pkl
- binance_1m_ohlcv/ETHUSDT_1m_ohlcv.pkl
- binance_1m_ohlcv/SOLUSDT_1m_ohlcv.pkl
- binance_1m_ohlcv/XRPUSDT_1m_ohlcv.pkl
- binance_1m_ohlcv/binance_all_1m_ohlcv.pkl

각 행의 의미:
- 각 행은 특정 자산의 1분 구간 하나를 의미한다.
- 예를 들어 BTCUSDT의 2026-02-19 00:10:00 UTC 행은
  00:10:00부터 00:10:59.999...까지의 1분 OHLCV 정보를 담는다.


1. 컬럼 설명
------------

[asset]
- 프로젝트에서 사용하는 자산 이름이다.
- 예:
  - bitcoin
  - ethereum
  - solana
  - xrp

[symbol]
- Binance 거래쌍 이름이다.
- 예:
  - BTCUSDT
  - ETHUSDT
  - SOLUSDT
  - XRPUSDT
- 모두 USDT 기준 spot market이다.

[interval]
- Binance candle 시간 간격이다.
- 현재 다운로드 코드는 1m, 즉 1분봉을 사용한다.

[slot_epoch]
- 해당 1분 candle의 시작 시각을 Unix timestamp 초 단위로 표현한 값이다.
- Polymarket pkl의 slot_epoch와 시간 기준으로 merge할 때 사용한다.

[open_utc]
- 해당 1분 candle의 시작 시각이다.
- UTC datetime 형식이다.

[close_utc]
- 해당 1분 candle의 종료 시각이다.
- UTC datetime 형식이다.

[open]
- 해당 1분 구간에서 첫 거래 가격이다.
- 시가라고 부른다.

[high]
- 해당 1분 구간에서 가장 높게 거래된 가격이다.
- 고가라고 부른다.

[low]
- 해당 1분 구간에서 가장 낮게 거래된 가격이다.
- 저가라고 부른다.

[close]
- 해당 1분 구간에서 마지막 거래 가격이다.
- 종가라고 부른다.

[volume]
- 해당 1분 구간 동안 거래된 base asset 수량이다.
- 예:
  - BTCUSDT라면 BTC 거래량
  - ETHUSDT라면 ETH 거래량
  - SOLUSDT라면 SOL 거래량
  - XRPUSDT라면 XRP 거래량

[return]
- 직전 1분봉 close 대비 현재 1분봉 close의 수익률이다.
- 계산식:
  return = close.pct_change()
- 예:
  - return = 0.001이면 직전 1분 종가 대비 약 0.1% 상승
  - return = -0.001이면 직전 1분 종가 대비 약 0.1% 하락

[quote_asset_volume]
- 해당 1분 구간 동안 거래된 quote asset 기준 거래대금이다.
- 현재 symbol들이 모두 USDT pair이므로 대략 USDT 기준 거래대금으로 볼 수 있다.

[number_of_trades]
- 해당 1분 구간 동안 발생한 거래 체결 횟수이다.

[taker_buy_base_volume]
- 해당 1분 구간 동안 taker buy로 체결된 base asset 수량이다.
- 매수자가 시장가성 주문으로 체결한 base asset 거래량에 가깝다.

[taker_buy_quote_volume]
- 해당 1분 구간 동안 taker buy로 체결된 quote asset 기준 거래대금이다.
- USDT pair에서는 taker buy 쪽 USDT 거래대금으로 볼 수 있다.


2. Polymarket 데이터와 연결하는 방식
-----------------------------------

Polymarket pkl의 한 행은 하나의 Up/Down 시장이다.
예를 들어 15분봉 시장이라면 slot_epoch부터 15분 동안의 시장을 의미한다.

Binance 1m OHLCV는 이 구간 안의 원물 가격 움직임을 제공한다.

예:
- Polymarket 5분 시장:
  - slot_epoch부터 slot_epoch + 5분 전까지의 Binance 1m candle 5개 사용

- Polymarket 15분 시장:
  - slot_epoch부터 slot_epoch + 15분 전까지의 Binance 1m candle 15개 사용

- Polymarket 1시간 시장:
  - slot_epoch부터 slot_epoch + 60분 전까지의 Binance 1m candle 60개 사용

이렇게 하면 Polymarket의 price_history_yes, price_history_no와
Binance 원물 OHLCV를 같은 1분 단위로 맞출 수 있다.


3. 현재 데이터에서 만들 수 있는 기술적 지표
------------------------------------------

open, high, low, close, volume만 있으면 다음 지표들을 계산할 수 있다.

- return
- log return
- rolling mean
- rolling volatility
- EMA
- MACD
- RSI
- Bollinger Band
- ATR
- VWAP
- OBV
- price range = high - low
- candle body = close - open
- upper/lower wick
- volume moving average
- taker buy ratio


4. 계산 가능한 기술적 지표를 입력 feature로 넣는 것이 의미가 있는가?
---------------------------------------------------------------

결론:
- 의미가 있을 수 있다.
- 다만 반드시 넣어야 하는 것은 아니다.
- 실험적으로 "raw OHLCV만 사용"과 "OHLCV + 기술적 지표 사용"을 비교하는 것이 가장 좋다.

기술적 지표는 open, high, low, close, volume으로부터 계산되는 값이므로
정보 이론적으로는 원본 데이터에 없는 완전히 새로운 정보를 추가하지 않는다.

하지만 모델 학습 관점에서는 여전히 유용할 수 있다.

이유 1: 모델이 직접 계산해야 할 패턴을 미리 제공한다.
- 예를 들어 RSI, MACD, EMA는 모두 과거 가격의 특정 변환이다.
- DQN이 raw OHLCV만 보고 이런 패턴을 스스로 학습할 수도 있지만,
  데이터가 많지 않거나 모델이 작으면 잘 못 배울 수 있다.
- 기술적 지표를 넣으면 사람이 만든 요약 feature를 제공하는 효과가 있다.

이유 2: 작은 데이터에서는 inductive bias가 도움이 된다.
- 현재 Polymarket 데이터는 수만 행 수준이다.
- 금융 시계열 딥러닝 기준으로 아주 큰 데이터는 아니다.
- 이럴 때는 raw data만 넣는 것보다 검증된 요약 지표가 학습 안정성을 높일 수 있다.

이유 3: DQN은 reward가 sparse할 수 있다.
- 이 프로젝트의 Phase 1 reward는 terminal reward 중심이다.
- 중간 보상이 적으면 학습 신호가 약하다.
- 상태 feature가 더 해석 가능한 형태이면 학습이 쉬워질 수 있다.

이유 4: baseline 모델과 비교하기 좋다.
- Logistic Regression, XGBoost 같은 baseline은 기술적 지표를 넣었을 때 성능이 좋아질 수 있다.
- DQN도 같은 feature set으로 비교하면 실험 설계가 깔끔해진다.


5. 기술적 지표를 넣을 때의 단점
-------------------------------

[중복 정보 증가]
- OHLCV에서 계산되는 값이므로 feature 간 상관관계가 커진다.
- 모델이 불필요하게 복잡해질 수 있다.

[과적합 위험]
- 지표를 너무 많이 넣으면 train 구간에만 맞는 패턴을 학습할 수 있다.
- 특히 금융 데이터는 regime change가 있어서 과적합에 주의해야 한다.

[look-ahead bias 위험]
- 기술적 지표 계산 시 미래 데이터를 섞으면 안 된다.
- 각 시점의 지표는 반드시 그 시점까지의 과거 데이터만 사용해야 한다.
- rolling, EMA, RSI 등을 계산할 때 centered window를 쓰면 안 된다.

[시간 정렬 문제]
- Polymarket 시장의 진입 시점에서 아직 알 수 없는 candle close를 사용하면 안 된다.
- 예를 들어 12:00에 의사결정한다면 12:00~12:01 candle의 close는 아직 모르는 정보이다.
- 환경을 만들 때 action 시점과 feature 시점을 엄밀히 맞춰야 한다.


6. 추천 실험 방식
-----------------

처음부터 모든 기술적 지표를 넣기보다는 단계적으로 비교하는 것이 좋다.

[Feature Set A: 최소 입력]
- Polymarket price_history_yes
- Polymarket price_history_no
- Polymarket volume
- Binance 1m return
- Binance 1m volume
- remaining time ratio
- portfolio state

[Feature Set B: OHLCV 추가]
- Feature Set A
- open
- high
- low
- close
- volume

[Feature Set C: 기술적 지표 추가]
- Feature Set B
- RSI
- MACD
- EMA
- Bollinger Band
- ATR
- VWAP
- OBV

이후 성능을 비교한다.
- A가 가장 좋으면 단순한 feature가 충분한 것이다.
- C가 더 좋으면 기술적 지표가 모델 학습에 도움이 된 것이다.
- C가 train에서는 좋고 test에서는 나쁘면 과적합 가능성이 크다.


7. 현재 프로젝트에 대한 추천
----------------------------

현재 단계에서는 다음 방식을 추천한다.

1. 먼저 Binance 1m OHLCV를 Polymarket 데이터와 정확히 merge한다.
2. return, volume, price range 정도의 단순 feature부터 사용한다.
3. DQN 환경이 제대로 작동하는지 확인한다.
4. 이후 RSI, MACD, EMA, ATR 같은 기술적 지표를 추가해 성능을 비교한다.

즉 기술적 지표는 "반드시 필요한 원천 데이터"라기보다는
"모델이 더 쉽게 학습하도록 돕는 가공 feature"로 보는 것이 적절하다.
